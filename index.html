Non-Negotiable Risk Rules
The rules that
protect the capital.

Every rule below is enforced in code — not willpower. The system refuses to place a trade that violates these rules. This is not discipline. This is engineering.

// Rule 01
Maximum 1% risk per trade

Never risk more than 1% of account balance on a single trade. On a $1,000 account that is $10 maximum loss per trade. Position size is calculated automatically from stop distance and account balance. The system refuses to place a trade that risks more.

position_size = (account * 0.01) / (entry - stop_loss)
// Rule 02
Maximum 5% daily drawdown → system pauses

If the account drops 5% in a single day, the system sends an alert, logs the event, and stops placing new trades until manually restarted. This prevents the most dangerous trading behaviour — doubling down after losses.

if daily_drawdown >= 0.05: system.pause() · notify(dad)
// Rule 03
Minimum 2:1 reward-to-risk ratio

Every trade must have a take profit at least twice the distance of the stop loss. A 30-pip stop requires a 60-pip target minimum. This means the strategy is profitable at a win rate as low as 34%. Below this ratio, the trade is not taken regardless of signal strength.

trade_valid = (take_profit - entry) / (entry - stop_loss) >= 2.0
// Rule 04
Maximum 3 open trades simultaneously

Never more than 3 positions open at once. Correlation check ensures no two highly correlated pairs trade simultaneously in the same direction. This prevents the illusion of diversification while actually holding the same risk twice.

if len(open_positions) >= 3: signal.ignore()
// Rule 05
No trading during high-impact news events

The system checks the economic calendar before placing a trade. If a high-impact news event (NFP, Fed rate decision, ECB press conference) is within 2 hours, no new positions are opened. News events invalidate technical signals.

if hours_to_news_event(impact='HIGH') < 2: signal.ignore()
// Rule 06
Weekly review with Dad as risk officer

Every Sunday: review the week's trades with Dad. Log what worked, what did not, and why. Any system changes require Dad's approval before going live. The 30-year software engineer is the second pair of eyes that prevents both bugs and behavioural errors.

weekly_review = mandatory · system_changes = dad_approved_only
